# AGENTS.md

This file provides guidance when working with code in this repository.

## What This Is

Scalar Sandbox — a full-stack app for sharing and previewing OpenAPI documents. Vue 3 frontend with Monaco editor + Scalar API Reference viewer, backed by Cloudflare Workers (Pages Functions) and D1 (SQLite).

## Commands

```bash
pnpm install              # Install dependencies
pnpm db:migrate           # Apply migrations to local D1 (run before first dev)
pnpm dev                  # Run frontend (Vite) + backend (Wrangler) concurrently
pnpm frontend             # Vite dev server only
pnpm backend              # Wrangler Pages dev server on port 8788
pnpm build                # Type-check (vue-tsc) then Vite build
pnpm format               # Format with Prettier
pnpm format:check         # Check formatting
pnpm db:generate          # Generate Drizzle migrations from db/schema.ts
pnpm db:migrate:production # Apply migrations to production D1
```

No test framework is configured.

## Architecture

**Not a monorepo.** Single package, full-stack:

- `src/` — Vue 3 SPA. Nearly all app logic lives in `src/App.vue` (editor state, share flow, data fetching); other components are mostly presentational.
- `functions/` — Cloudflare Pages Functions (serverless API). Each file exports `onRequest`; the D1 binding is `context.env.DB`, wrapped with Drizzle via `drizzle(context.env.DB)`.
  - `api/share.ts` — POST to create a new spec, or update an existing one when a `?token=` query param is present
  - `api/share/[id].ts` — GET raw spec row by ID (used by the frontend to load content)
  - `files/[[id]].ts` — catch-all; `context.params.id` is `[id, view]`. Serves `openapi.json`, `openapi.yaml`, or `openapi.yml`, normalized through `@scalar/openapi-parser`
- `db/schema.ts` — Drizzle ORM schema (single `Specs` table on D1/SQLite)
- `migrations/` — SQL migrations generated by Drizzle Kit

**Dev proxy:** Vite proxies `/api/*` and `/files/*` to Wrangler on port 8788 during development. Both `pnpm frontend` and `pnpm backend` must run for the app to work (`pnpm dev` runs both).

## Data Model & Share Flow

The `Specs` table is effectively **append-only versioned**: `id` (nanoid, 5 chars), `parentId`, `content`, `token` (32-char hex secret), `createdAt`.

- Saving in the UI (`share()` in `App.vue`) POSTs to `/api/share` with the previous spec's `id` as `parentId`, creating a **new** row and a new URL each time — it does not mutate the old one. The `parentId` chain records edit lineage.
- The `token` supports in-place updates (`POST /api/share?token=...`), but the current frontend does not use it — it always creates new versions.
- **No auth.** Anyone with an ID can read a spec; the `token` is the only write secret.

## Routing

Vue Router (`src/router.ts`) maps three paths — all render `App.vue`:
- `/` — blank editor
- `/e/:id` — edit mode (`name: 'edit'`)
- `/p/:id` — preview mode (`name: 'preview'`)

Toggling edit/preview in the UI swaps between these routes; loading an `:id` fetches its content from `/api/share/:id`.
